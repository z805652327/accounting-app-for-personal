# 个人记账 App — 会计逻辑设计方案

## 核心原则

基于借贷记账法，适配个人场景，自动产出三张财务报表（资产负债表、利润表、现金流量表）。

### 三级明细（统一三级层级）

所有科目下统一使用**三级明细**作为最细粒度分类。不论收入/费用还是资产/负债，二级科目下面都是三级明细，统一使用 **8 位编码**。

| 科目范围 | 三级明细是什么 | 示例 |
|---------|-------------|------|
| 收入类(4xxx)、费用类(5xxx) | 消费/收入明细分类 | `50101001` 阳光花园房租 |
| 资产类(1xxx)  | 每个资金容器/资产/资产备抵项 | `10201001` 招商银行1111 / `15103001` 联想笔记本折旧 |
| 负债类(2xxx)  | 每笔负债/欠款 | `20100001` 招商银行2222 |

三级明细由系统自动分配 8 位编码，用户不可手动编辑。

**编码规则**：`PPPPPSSS` — 父级编码(补齐5位, 末位补0) + 3位流水号

| 父级 | 规则 | 示例三级编码 |
|-----|------|------------|
| 10201（5位） | 直接拼接 | `10201001`、`10201002` |
| 101（3位） | 末位补0至5位 → `10100` | `10100001`、`10100002` |
| 50101（5位） | 直接拼接 | `50101001`、`50101002` |

**两种三级明细的差异**（用户不必关注，仅系统层面区分）：

| 维度 | 分类型三级明细（收入/费用） | 容器型三级明细（资产/负债） |
|-----|------------------------|------------------------|
| 存储表 | `accounting_subjects` 表 | `accounts` 表（即账户实例） |
| 创建方式 | 记账时直接输入新名称，或科目管理集中维护 | 在账户管理页面创建，有独立余额/生命周期 |
| 编码字段 | `code`（科目编码） | `subject_code`（账户编码） |
| 余额 | 无余额，纯分类标记 | 有独立余额，随交易增减 |
| 同一条交易 | 有值（区分消费类别），同时账户列也有值（付款账户） | 有值（付款/收款容器），同时三级列也有值（消费类别） |

**创建入口**：
- 分类型三级明细：记账时科目选择器未找到合适分类 → 直接输入新名称 → 系统自动创建并分配编码
- 容器型三级明细：在账户管理页面新增，需选择所属二级科目
- 两者均可在「设置 → 科目管理」中集中管理

**停用策略**：
- 三级明细可停用（非删除），停用后已有交易不受影响，但新交易不能再选择该明细
- 停用的三级明细在科目选择器中隐藏，在科目管理中标记为「已停用」
- 已停用的三级明细可重新启用，编码不变

**科目余额表展示**（统一展开到三级明细）：

```
活期账户(10201)
├── 10201001 招商银行1111          ¥50,000
├── 10201002 建设银行2222          ¥30,000

├── 50101001 阳光花园房租          ¥2,000

```

## 二、重点场景的会计处理

### 场景 1：押一付三（预付摊提）

```
场景：租房押一付三，月租 2,000，共付 8,000
  押金 2,000 → 押金/保证金(142)，可收回
  当月租金 2,000 → 房租(50101)，立即确认费用
  预付租金 4,000 → 待摊费用(141)，分 2 个月摊销

支付时分录：
  借：房租(50101)              2,000  ← 当月费用
  借：待摊费用(141)            4,000  ← 预付第 2-3 月租金
  借：押金/保证金(142)         2,000  ← 可回收押金
  贷：银行存款(102)            8,000

第 2 个月摊销：
  借：房租(50101)              2,000
  贷：待摊费用(141)            2,000

第 3 个月摊销：
  借：房租(50101)              2,000
  贷：待摊费用(141)            2,000
```

### 场景 2：固定资产折旧

```
实际场景：花 10,000 元买了台电脑（联想笔记本），预计使用 3 年，残值 1,000

用户操作：
  类型=购置固定资产 | 资产=联想笔记本（三级明细）| 分类=电子设备(13103) | 账户=银行卡
  金额=10,000 | 折旧年限=3年(36月) | 残值=1,000

购买时分录：
  借：电子设备(13103)        10,000  ← 原值始终保留
  贷：银行存款(102)          10,000

每月自动生成折旧（直线法）：
  月折旧额 = (10,000 - 1,000) ÷ 36 ≈ 250

  借：资产折旧(50701)              250
  贷：15103001 联想笔记本折旧      250  ← 贷记该资产的三级明细折旧科目

3年后账面净值 = 10,000 - 250×36 = 1,000 ✓ 等于残值

查原值：电子设备(13103) = 10,000
查累计折旧：15103001 联想笔记本折旧 = 9,000
查净值：10,000 - 9,000 = 1,000
```
### 场景 3：股票/基金投资（成本法 + 可选估值调整）

这是最核心的设计决策，我建议采用**成本法为主，定期估值调整为辅**的策略：

#### 3.1 买入（成本法）
```
场景：10,000 元买入贵州茅台
用户操作：
  类型=投资买入 | 投资品=贵州茅台（三级明细）| 账户=银行卡
  名称=贵州茅台 | 单价=100 | 总金额=10,000

分录：
  借：股票(12101)       10,000
  贷：银行存款(102)          10,000

持有期间 → 始终按成本 10,000 计量，不跟随每日涨跌
```

#### 3.2 分红
```
场景：持仓的贵州茅台分红 500 元到银行卡

用户操作：
  类型=收入 | 分类=分红收入(40203) | 账户=银行卡 | 投资品=贵州茅台（三级明细）
  金额=500

分录：
  借：银行存款(102)            500
  贷：分红收入(40203)          500
```

#### 3.3 定期估值调整（可选，用户按需操作）
```
场景：月底看持仓，贵州茅台市值从 10,000 涨到 12,000

用户操作：
  类型=估值调整 | 投资品=贵州茅台（三级明细）| 账户=不涉及资金
  原值=10,000 | 最新市值=12,000

分录：
  借：股票(12101)              2,000
  贷：公允价值变动储备(303)   2,000  ← 直接进净资产，不影响当月利润
```

> **为什么增值不进利润表？** 个人场景下，未实现浮盈/浮亏进利润表会导致利润剧烈波动。公允价值变动储备作为净资产的组成部分，只在卖出时才将已实现部分转入利润表。

#### 3.4 卖出并确认收益
```
场景：卖出全部贵州茅台持仓，得 13,000 到银行卡（成本 10,000，已估值调增 2,000）

用户操作：
  类型=投资卖出 | 投资品=贵州茅台（三级明细）| 账户=银行卡
  卖出金额=13,000 | 该笔原值=10,000

系统处理：
  ① 转回该投资品的公允价值变动储备：
     借：公允价值变动储备(303)    2,000
     贷：股票(12101)              2,000  ← 估值调增回拨
     此时贵州茅台三级明细账面回到 10,000
  
  ② 确认卖出：
     借：银行存款(102)           13,000
     贷：股票(12101)            10,000
     贷：已实现投资收益(40201)     3,000  ← 收益进入利润表
```

**总结投资逻辑**：

| 操作 | 资产负债表影响 | 利润表影响 | 现金流 |
|------|--------------|-----------|--------|
| 买入 | 现金↓ 投资↑ | 无 | 投资流出 |
| 分红 | 现金↑ | 投资收益↑ | 投资流入 |
| 估值上调 | 投资↑ 储备↑ | 无 | 无 |
| 估值下调 | 投资↓ 储备↓ | 无 | 无 |
| 卖出 | 现金↑ 投资↓ | 实现收益↑ | 投资流入 |

#### 3.5 部分卖出（平均成本法）

```
场景：持有贵州茅台 200 股，总成本 20,000（均价 100/股），已估值调增 4,000
      卖出 100 股，每股售价 150，收款 15,000 到银行卡

用户操作：
  类型=投资卖出 | 投资品=贵州茅台（三级明细）| 账户=银行卡
  卖出数量=100 | 卖出单价=150 | 卖出金额=15,000

系统处理：
  ① 计算：
     卖出比例 = 100 ÷ 200 = 50%
     转出成本 = 20,000 × 50% = 10,000
     转出估值调增 = 4,000 × 50% = 2,000

  ② 转回对应比例的估值调整：
     借：公允价值变动储备(303)    2,000
     贷：股票(12101)              2,000  ← 对应比例估值回拨

  ③ 确认卖出：
     借：银行存款(102)           15,000
     贷：股票(12101)            10,000
     贷：已实现投资收益(40201)     5,000  ← 收益 = 15,000 - 10,000

  ④ 更新持仓：
     剩余 100 股，剩余成本 10,000，剩余估值储备 2,000
```

**为什么不追踪每日净值？**
- 用户无法每天更新所有持仓市值
- 每日波动对个人记账没有实际意义
- 成本法 + 定期估值调整是在"准确"和"可用"之间的最佳平衡

### 场景 4：房贷月供（本金利息拆分）

```
场景：房贷月供 5,000，其中利息 3,000，本金 2,000

用户操作：
  类型=还款 | 账户=银行卡 | 负债=房贷(22101)
  还款总额=5,000 | 其中利息=3,000 | 本金=2,000

分录：
  借：利息支出(50801)          3,000  ← 费用进利润表
  借：长期借款-房贷(22101)     2,000  ← 负债减少
  贷：银行存款(102)            5,000  ← 现金减少
```

### 场景 5：发工资代扣社保个税

```
场景：月薪 10,000，社保公积金扣 2,250，个税扣 90，实发 7,660

用户操作：
  类型=发工资 | 分类=工资薪金(40101) | 账户=银行卡
  税前总额=10,000 | 个税=90 | 社保公积金=2,250

分录（简化版 — 所有扣项均记为支出）：
  借：银行存款(102)             7,660  ← 实发到手
  借：个人税费(50803)              90  ← 个税
  借：社保及公积金(50804)       2,250  ← 养老+医疗+失业+公积金
  贷：工资薪金(40101)         10,000  ← 全额税前工资

利润表体现：
  工资薪金          10,000  ← 收入
  个人税费              90  ← 支出
  社保及公积金       2,250  ← 支出
  本期结余           7,660
```

> **说明**：社保公积金中的养老/医疗个人账户部分理论上属于个人资产，但简化版全额记为支出。优点是无需追踪社保账户余额，对个人记账更实用。

### 场景 6：资产处置与报废

```
场景：电脑原值 10,000，已提折旧 6,000（24个月），账面净值 4,000
      对应折旧三级明细：15103001 联想笔记本折旧

▸ 情况 A：二手卖出得 3,000（亏损 1,000）

  用户操作：
    类型=资产处置 | 资产=联想笔记本（三级明细）| 收款账户=银行存款
    售价=3,000 | 处置原因=二手卖出

  系统分录：
    借：银行存款(102)              3,000  ← 售价入账
    借：15103001 联想笔记本折旧    6,000  ← 转销该资产已提折旧
    借：资产处置损益(50703)       1,000  ← 亏损（售价 < 净值）
    贷：电子设备(13103)          10,000  ← 转销资产原值

▸ 情况 B：二手卖出得 5,000（盈利 1,000）

  系统分录：
    借：银行存款(102)              5,000
    借：15103001 联想笔记本折旧    6,000
    贷：电子设备(13103)          10,000
    贷：资产处置损益(50703)       1,000  ← 盈利（红字/贷方余额冲减费用）

▸ 情况 C：直接报废（无残值收入）

  用户操作：
    类型=资产处置 | 资产=联想笔记本（三级明细）| 无收款账户
    处置原因=报废

  系统分录：
    借：15103001 联想笔记本折旧    6,000
    借：资产处置损益(50703)       4,000  ← 全部未折旧部分记为损失
    贷：电子设备(13103)          10,000
```

> 处置资产时自动转销该资产的原值和累计折旧，差额计入 50703 资产处置损益。盈利时 50703 出现贷方余额，在利润表中表现为费用减项（即变相收入）。

## 五、个人财务指标体系

### 5.1 偿债能力指标

| 指标 | 公式 | 健康参考 | 说明 |
|------|------|---------|------|
| 资产负债率 | 总负债 ÷ 总资产 | < 50% | 负债水平是否安全 |
| 流动比率 | (现金+存款+货基+应收) ÷ 流动负债 | > 1.5 | 短期偿债能力 |
| 紧急备用金月数 | (现金+存款+货基) ÷ 月均固定支出 | 3~6 个月 | 失业/急事能撑多久 |

### 5.2 结余能力指标

| 指标 | 公式 | 健康参考 | 说明 |
|------|------|---------|------|
| 月结余率 | 本月结余 ÷ 本月收入 | > 30% | 赚的钱存下了多少 |
| 投资回报率 | (已实现收益+分红) ÷ 投资成本 | 视市场 | 投资表现评估 |
| 财务自由度 | 投资收入 ÷ 月均总支出 | 100%=自由 | 被动收入覆盖支出的比例 |

### 5.3 支出结构指标

| 指标 | 公式 | 健康参考 | 说明 |
|------|------|---------|------|
| 恩格尔系数 | 餐饮支出 ÷ 总支出 | < 30% 富裕 | 吃占比越低生活水平越高 |
| 固定支出占比 | 固定支出 ÷ 总支出 | < 50% | 越低则财务灵活性越好 |
| 弹性支出占比 | 弹性支出 ÷ 总支出 | — | 理论上可压缩的支出空间 |

### 5.4 综合指标

| 指标 | 公式 | 说明 |
|------|------|------|
| 净资产增长率 | (期末净资产 - 期初净资产) ÷ 期初净资产 | 整体财务增速 |
| 总资产增长率 | (期末总资产 - 期初总资产) ÷ 期初总资产 | 财富积累速度 |

### 5.5 自定义指标

**用户可基于公式创建自己的指标：**

```
可用元素：
  - 科目编码范围：SUM(502) = 502类汇总，SUM(50201) = 单个科目汇总，SUM(50201001) = 三级明细粒度
  - 运算符：+, -, *, /, (), AVG
  - 系统变量：TOTAL_INCOME, TOTAL_EXPENSE, NET_ASSET

示例：
  "宠物支出占比" = SUM(50301) / TOTAL_EXPENSE       ← 前提：用户用50301记宠物开销
  "税负率" = SUM(50803) / TOTAL_INCOME
  "交通总成本" = SUM(50202) + SUM(50404)             ← 日常通勤+旅游交通合计
```

> 自定义指标存储在 `user_indicators` 表中，公式支持科目编码引用。系统在每月报表生成时一并计算。

## 八、首次启动引导

### 8.1 初始化流程

首次安装打开 App 时，引导用户填写初始化数据：

```
① 添加资产账户
   用户输入：银行卡余额 50,000 / 现金 2,000 / ...

② 添加已有负债（可选跳过）
   用户输入：房贷余额 800,000 / 信用卡欠款 3,000 / ...

③ 系统自动计算
   期初净资产 = 总资产 - 总负债
   → 例：52,000 - 803,000 = -751,000（负数说明负债＞资产）

④ 生成初始化分录
   借或贷：各账户和负债科目 → 差额计入期初净资产(301)
```

### 8.2 初始化后仍可随时调整

不强制用户一次性填写完整：

- 页面上提供「跳过引导，先进 App」按钮
- 设置页提供「资产初始化」入口，可随时补充/调整期初数据

**调整方式（差额调整分录）**：

用户修改某项期初余额后，系统不直接修改历史数据，而是自动生成一笔调整分录，将差异记入当期：

```
示例：用户发现上月填的招商银行1111余额少了 ¥5,000

用户操作：
  进入「资产初始化」→ 招商银行1111 → 将余额从 50,000 修改为 55,000

系统处理：
  计算差额 = 55,000 - 50,000 = +5,000
  自动生成一笔当期分录：
    借：银行存款(102)            5,000
    贷：期初净资产(301)           5,000
```

| 调整方向 | 分录 |
|---------|------|
| 资产类余额调增 / 负债类余额调减 | 借：对应科目 → 贷：期初净资产(301) |
| 资产类余额调减 / 负债类余额调增 | 借：期初净资产(301) → 贷：对应科目 |

**效果**：
- 历史报表不受影响（已归档月份不变）
- 差异通过当期调整分录体现，所有交易流水可追溯
- 系统在生成调整分录前展示差额预览，用户确认后执行

## 九、数据校验与对账

### 9.1 自动校验机制

每次交易后，系统自动计算并校验会计恒等式：

```
总资产 = SUM(资产类科目余额)
总负债 = SUM(负债类科目余额)
总净资产 = SUM(净资产类科目余额) + SUM(收入类科目余额) - SUM(费用类科目余额)

校验：总资产 = 总负债 + 总净资产
```

- 校验通过：无提示
- 校验不通过：首页顶部出现黄条警告「账目不平，差额 ¥X,XXX」，不阻断操作
- 用户点击黄条进入「对账助手」

### 9.2 对账助手

当账不平或用户想核验时，提供：

| 功能 | 说明 |
|------|------|
| 科目余额总览 | 列出所有科目当前余额，用户逐一核对 |
| 差额提示 | 显示总资产 - 总负债 - 总净资产的差额 |
| 常见原因排查 | 检查是否存在孤立交易（缺少对应的借贷科目） |
| 手动更正 | 用户可录入一笔调整分录来修正平衡 |
| 导出对账底稿 | 导出 Excel 供外部核验 |

### 9.3 账户摘要与属性

每个账户支持查看「账户摘要」：

```

│ 期初余额：¥50,000               │

│ 当前余额：¥57,000               │
│ 最近交易：5笔                   │

└───────────────────────────────┘
```

账户属性校验：

| 项目 | 说明 |
|------|------|
| 计提周期 | 按月计提 |

### 10.1 折旧配置修改规则

| 场景 | 处理方式 |
|------|---------|

## 十一、OCR 识别

### 11.1 实现方案

使用 **tesseract.js**（纯前端 OCR 引擎，通过 WebAssembly 运行）：

```
用户拍照/选图 → 获取图片 base64
  → tesseract.js 加载 WASM 引擎 + 中文语言包（约 13MB）
  → 在浏览器/WebView 内本地完成识别（不上传任何图片）
  → 返回识别文本
  → 正则提取：金额、日期、商家名称
  → 预填记账表单（金额/日期自动填入，科目由**用户手动选择三级明细**）
```

### 11.2 关键参数

| 参数 | 值 |
|------|-----|
| 引擎 | Tesseract.js 5.x（WASM） |
| 语言包 | chi_sim（简体中文）+ eng（数字） |
| 首次加载 | ~13MB（引擎 8MB + 语言包 5MB） |
| 加载策略 | 首次使用 OCR 时提示下载，非强制预装 |
| 识别速度 | PC 1~3 秒，手机 2~5 秒 |
| 识别率 | 印刷体小票/发票 ~90%，手写 ~50% |
| 图片要求 | 建议 300dpi 以上，最低 150dpi |

### 11.3 提取字段

从 OCR 文本中通过正则提取：

| 字段 | 提取方式 | 示例 |
|------|---------|------|
| 金额合计 | 匹配"合计/总额/总计/¥"等关键词后数字 | ¥128.00 |
| 交易日期 | 匹配 YYYY-MM-DD 或 YYYY年MM月DD日 | 2026-05-19 |
| 商家名称 | 提取发票标题或收款方名称 | 星巴克咖啡 |
| 商品明细 | 提取商品清单行（可选） | 拿铁 × 1 |

| 功能 | 说明 |
|------|------|
| 锁定期限 | 交易发生后超过 **2 个月（60 天）**自动归档锁死 |
| 锁定后操作 | 无法修改、无法删除 |
| 锁定前提醒 | 交易接近 60 天时（如第 55 天），在交易列表显示"即将归档"标识 |
| 查看不受影响 | 锁死不影响查看交易详情和参与报表计算 |
| 设计考虑 | 2 个月的窗口期足够用户核对当月账目。锁死后保证历史数据的稳定性和报表的一致性 |

### 13.7 分录号与操作记录的关联

| 项目 | 决策 |
|------|------|
| 分录号体系 | **不引入**分录号（凭证号）体系 |
| 操作记录关联 | 不使用分录号关联，编辑历史直接记录 `transaction_id` + 变更前后字段值 |
| 原因 | ① 个人场景无需审计级追溯，编辑历史已足够查账 ② 一笔交易可能生成多条分录，引入分录号会使关联关系变成 N:N，复杂度急剧上升 ③ 编辑历史 + 归档锁死的组合已覆盖"谁在什么时候改了什么"的核心需求 ④ 减少 3-5 张关联表和对应的维护代码 |
| 替代方案 | 编辑历史表中直接增加 `change_reason` 字段（如"用户编辑"、"系统自动折旧"、"系统自动摊销"），区分人工与系统操作，无需引入分录号体系 |

### 13.8 年末结账（可选）

| 功能 | 说明 |
|------|------|
| 触发方式 | 用户主动发起（设置 → 账务管理 → 年末结账），系统不自动执行 |

| 不结转 | 用户可选择不执行年末结账，302 持续累加，不影响任何功能。任何时候可再发起结转 |

### 16.2 实体冲突处理

导入文件中的实体（科目、账户、预算、标签、自定义指标、自定义报表）与本地数据冲突时的处理策略：

| 冲突类型 | 处理方式 | 用户操作 |
|---------|---------|---------|
| **科目编码重复** | 导入的科目编码与本地已有科目相同但名称不同 | 弹出选择：① 合并到本地科目 ② 跳过该科目 ③ 映射到其他科目（用户选一个本地科目接收数据） |
| **账户名称重复** | 导入的账户名与本地账户同名 | 弹出选择：① 合并（追加交易到本地账户）② 跳过 ③ 映射到其他账户 |
| **标签重复** | 导入的标签名与本地标签同名 | 自动合并（标签无结构冲突） |
| **预算冲突** | 导入的预算与本地同一科目同一期间的预算冲突 | 弹出选择：① 覆盖为导入值 ② 保留本地值 ③ 跳过 |
| **自定义指标/报表** | 导入的公式或名称与本地冲突 | 弹出选择：① 覆盖 ② 保留本地 ③ 跳过 |

> 用户可在导入开始时统一预设冲突策略（全部合并 / 全部跳过 / 逐项确认），默认逐项确认。

### 16.3 导入流程

```
用户选择 JSON 文件
  → 解析文件，校验格式
  → 检测交易重复 + 实体冲突
  → 展示冲突清单（交易 + 实体分页展示）
  → 用户逐项或批量处理冲突
  → 确认导入
  → 写入数据
  → 展示导入报告（新增/跳过/失败明细）
```

### 新增字段/表

1. **`accounting_subjects`** 表新增字段：

   - `code` 字段从 5 位扩展到 8 位 — 三级明细编码格式 `PPPPPSSS`（父级5位 + 流水3位）
   - `level`: `TINYINT` — 层级（1=一级, 2=二级, 3=三级）
   - `parent_id`: 上级科目 ID（一级为 null，二级指向一级，三级指向二级）
   - 注：系统预设科目（L1/L2）为只读不可删除，用户新增 L3 科目可编辑停用

2. **`transactions`** 表扩展：
   - `tx_type`: 上述 14 种交易类型
   - `subject_id`: 关联的会计科目（收入/费用/投资/负债分类），指向 L2
   - `l3_subject_id`: 关联的三级明细 ID（nullable），指向 accounting_subjects（分类型L3）或 accounts（容器型L3）。收入/费用类交易记录消费分类，资产/负债类交易可通过 account_id 定位容器，此时 l3_subject_id 可为空
   - `to_account_id`: 转账/投资的目的账户
   - `to_subject_id`: 关联的第二个科目

3. **`amortization_schedules`** 新增表 — 摊提计划
   - 关联 transaction_id
   - 关联 l3_subject_id（分摊到的三级明细）
   - 总金额、摊销期数、每期金额、剩余期数
   - 系统每月自动检查并生成摊提分录

4. **`depreciation_configs`** 新增表 — 折旧配置
   - 关联 asset_id（accounts 表中 account_type='fixed_asset' 的实例）
   - 原值、残值、使用月数、折旧方法（直线法）
   - 系统每月自动生成折旧分录
   - 使用累计折旧备抵科目（151xx），保留资产原值

5. **`investment_valuations`** 新增表 — 投资估值记录
   - 关联 subject_id
   - 估值日期、成本、市价、浮动盈亏

6. **`user_indicators`** 新增表 — 自定义指标
   - 用户 ID、指标名称、公式表达式、小数位数、是否启用
   - 支持引用科目编码（如 SUM(50202)）和系统变量
   - 每月报表生成时自动计算

7. **`saved_reports`** 新增表 — 自定义报表
   - 用户 ID、报表名称、筛选条件（JSON）、排序字段
   - 显示字段列表、是否置顶

8. **`accounts`** 表新增字段 — 账户实例扩展
   - `subject_code`: `VARCHAR(8)` — 三级明细编码，格式 `PPPPPSSS`（父级5位 + 流水3位），系统自动分配，唯一索引
   - `account_type`: `'cash' | 'checking' | 'fixed_deposit' | 'money_market' | 'receivable' | 'investment' | 'fixed_asset' | 'prepaid' | 'deposit' | 'credit_card' | 'payable' | 'loan'` — 区分账户类型，决定哪些扩展字段生效
   - `bank_name`: 银行名称，如"招商银行"
   - `card_last_four`: 卡号后四位，用于识别
   - `credit_limit`: 信用卡额度（仅 credit_card 类型有值）
   - `maturity_date`: 到期日（仅 fixed_deposit 类型有值）
   - `notes`: 备注/说明，投资品统称写在此处（如"贵州茅台 600519"），用户输入名称时自由填写代码
   - `contract_no`: 合同编号（仅 loan 类型有值）

9. **`budget_overall`** 新增表 — 月度总预算配置
   - 关联用户 ID
   - `monthly_cap`: 月度总支出上限
   - `active_month`: 生效年月（YYYYMM）
   - `is_active`: 是否启用

10. **`budget_thresholds`** 新增表 — 科目阈值配置
    - 关联用户 ID
    - `subject_id`: 关联的费用科目（支持 L2 或 L3，由用户选择粒度）
    - `amount`: 预警线金额
    - `is_active`: 是否启用

11. **`investment_lots`** 新增表 — 投资批次记录
    - 关联 subject_id（投资科目）
    - 关联 l3_subject_id（具体投资品三级明细）
    - `buy_date`: 买入日期
    - `quantity`: 买入数量/份额
    - `unit_cost`: 买入单价
    - 平均成本法：卖出时按加权均价计算成本

### 18.1 数据库迁移策略

#### 解决的问题
App 版本升级后，数据库结构可能变化（新增表/列），需要自动升级旧用户的数据表，避免数据丢失或 App 崩溃。

#### 实现方案

```sql
-- app_settings 表记录版本
INSERT INTO app_settings (key, value) VALUES ('db_version', '1');

-- App 启动时执行
current_version = get_db_version()    -- 从 app_settings 读取
while (current_version < CODE_VERSION):
    run_migration(current_version + 1)  -- 执行对应版本的 SQL 脚本
    current_version++
save_db_version(current_version)
```

#### 迁移示例

| 版本 | 变更内容 | SQL |
|------|---------|-----|
| v1 → v2 | transactions 表新增 tags 列 | `ALTER TABLE transactions ADD COLUMN tags TEXT;` |
| v2 → v3 | 新增 user_indicators 表 | `CREATE TABLE user_indicators (...);` |
| v3 → v4 | 新增分拆交易表 | `CREATE TABLE split_transactions (...);` |

#### 用户无感知

- 升级 App 后打开，自动执行迁移脚本
- 过程中显示"正在更新数据..."
- 迁移失败自动回滚，App 以旧版本逻辑运行

## 十九、应用锁（PIN码）

| 功能 | 说明 |
|------|------|
| 开启方式 | 设置 → 安全 → 开启应用锁 |
| 验证方式 | 6 位数字 PIN 码 |
| 首次设置 | 输入 PIN 码 → 再次确认 → 设置密保问题（可选，用于找回 PIN） |
| 锁定时机 | App 进入后台超过 **1 分钟**后重新打开时验证 |
| 解锁失败 | 连续 5 次错误 → 锁定 30 秒，10 次错误 → 锁定 5 分钟 |
| 关闭/修改 | 需输入原 PIN 码验证身份 |

### 导出时的 PIN 验证

| 场景 | 是否需要 PIN | 说明 |
|------|-------------|------|
| 打开 App | 是（如已开启） | 进入主界面需要解锁 |
| 导出 JSON 备份 | **是** | 备份包含全量财务数据，导出前需验证 |
| 导出 CSV/Excel | **是** | 导出交易明细需验证 |
| 导出报表 Excel | **是** | 导出报表需验证 |
| 导入 JSON 文件 | **否** | 导入操作已有数据冲突确认流程，不重复验证 |
| 数据迁移/清理操作 | **是** | 涉及数据变动的敏感操作需验证 |

> **跨端同步说明**：应用锁是纯本地功能，不影响 JSON 文件本身。导出文件时验证 PIN 确保操作者是本人，但导出的 JSON 文件不加密（用户可自行用其他工具加密）。PIN 码不保存在备份文件中。
